1. SSG (Static Site Generation)
Як працює: Сторінка генерується один раз під час збірки проєкту (npm run build). Всі користувачі отримують один і той самий готовий HTML-файл.

Переваги: Найшвидший варіант. Сторінки лежать на CDN.

Кейс: Блог, документація, сторінка "Про нас".

2. SSR (Server-Side Rendering)
Як працює: HTML генерується на сервері для кожного запиту індивідуально. Коли користувач заходить на сторінку, сервер "на льоту" збирає дані, рендерить сторінку і віддає її.

Переваги: Завжди актуальні дані, персоналізація (наприклад, кабінет користувача).

Кейс: Стрічка новин, сторінка з особистими налаштуваннями.

3. ISR (Incremental Static Regeneration)
Як працює: Це "гібрид". Ви створюєте статичну сторінку (SSG), але кажете Next.js: "Оновлюй її у фоні не частіше, ніж раз на 60 секунд".

Переваги: Швидкість як у SSG, але дані не застарівають на тижні.

Кейс: Каталог товарів (ціни можуть змінюватись, але нам не треба рендерити 10 000 сторінок для кожного клієнта окремо).

4. Використовуйте Клієнтські компоненти ('use client'), якщо:

Потрібна інтерактивність: onClick, onChange тощо.

Використовуються хуки стану або життєвого циклу: useState, useReducer, useEffect, useLayoutEffect.

Використовуються Browser APIs: window, document, localStorage, navigator.

Використовуються кастомні хуки, які всередині мають вищеперелічене.

Використовуйте Серверні компоненти (за замовчуванням), якщо:

Потрібно отримати дані з бази або API (Direct Data Fetching).

Потрібно зберігати конфіденційну інформацію (токени, ключі API), які не мають потрапити в браузер.

Компонент просто відображає дані і не має складної логіки взаємодії.

5. "Золоте правило" архітектури Next.js
На інтерв'ю часто питають про "Client Boundary" (межу клієнта).
Важливо розуміти: як тільки ви поставили 'use client' зверху файлу, всі компоненти, які імпортуються в цей файл, теж стають клієнтськими.

Порада для профі: Намагайтеся тримати 'use client' якомога нижче по дереву компонентів ("Leaf Components").

Погана практика: Робити всю сторінку клієнтською.

Гарна практика: Сторінка — серверна (бере дані), а маленька кнопка "Лайк" всередині неї — клієнтська.

6. Кешування

Ось три основні стратегії, які ви повинні знати для інтерв'ю:

1. Повне відключення кешу (Dynamic Data)
Якщо вам потрібно, щоб дані завантажувалися заново при кожному візиті користувача (як getServerSideProps у старому підході).

JavaScript
const response = await fetch('https://api.example.com/data', {
  cache: 'no-store'
});
Коли використовувати: Курси валют, кошик покупок, особисті повідомлення.

2. Часове оновлення (Incremental Static Regeneration - ISR)
Це той самий ISR, про який ми згадували раніше. Ви кажете Next.js: "Використовуй кеш, але перевір дані через N секунд".

JavaScript
const response = await fetch('https://api.example.com/data', {
  next: { revalidate: 60 } // оновлювати не частіше ніж раз на хвилину
});
Як це працює: Перший користувач бачить старі дані, але цей візит "будить" сервер, той робить запит у фоні, оновлює кеш, і другий користувач уже бачить нове.

Коли використовувати: Списки товарів, пости в блозі.

3. Теговане оновлення (On-demand Revalidation)
Це найбільш просунутий метод. Ви даєте запиту "ім'я" (тег). Кеш буде жити вічно, поки ви самі не скажете його видалити (наприклад, через адмін-панель).

JavaScript
// У компоненті
const response = await fetch('https://api.example.com/data', {
  next: { tags: ['posts'] }
});

// В іншому місці (наприклад, в API Route після створення нового посту)
import { revalidateTag } from 'next/cache';
revalidateTag('posts');
